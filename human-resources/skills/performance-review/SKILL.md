# /performance-review

> 익숙하지 않은 플레이스홀더가 보이거나 연결된 도구를 확인해야 한다면 [CONNECTORS.md](../../CONNECTORS.md)를 참고하세요.

성과 리뷰 템플릿을 생성하고 피드백 구조화를 돕습니다.

## 사용법

```
/performance-review $ARGUMENTS
```

## 모드

```
/performance-review self-assessment       # 자기평가 템플릿 생성
/performance-review manager [employee]    # 특정 인원에 대한 관리자 리뷰 템플릿
/performance-review calibration           # 캘리브레이션 준비 문서
```

모드가 지정되지 않으면 어떤 종류의 리뷰가 필요한지 물어봅니다.

## 출력 - 자기평가 템플릿

```markdown
## 자기평가: [Review Period]

### 주요 성과
[이번 기간의 주요 성과 3-5개를 나열하세요. 각 항목마다 상황, 본인의 기여, 영향력을 설명하세요.]

1. **[Accomplishment]**
   - Situation: [Context]
   - Contribution: [What you did]
   - Impact: [Measurable result]

### 목표 검토
| 목표 | 상태 | 근거 |
|------|--------|----------|
| [지난 기간의 목표] | 달성 / 초과 달성 / 미달성 | [근거] |

### 성장 영역
[어디서 성장했나요? 새로운 기술, 확장된 역할 범위, 리더십 발휘 순간.]

### 어려웠던 점
[무엇이 어려웠나요? 무엇을 다르게 했을까요?]

### 다음 기간 목표
1. [Goal — 구체적이고 측정 가능]
2. [Goal]
3. [Goal]

### 관리자에게 주는 피드백
[매니저가 어떻게 더 잘 지원할 수 있나요?]
```

## 출력 - 관리자 리뷰

```markdown
## 성과 리뷰: [Employee Name]
**기간:** [Date range] | **관리자:** [Your name]

### 전체 평가: [Exceeds / Meets / Below Expectations]

### 성과 요약
[2-3문장의 종합 평가]

### 핵심 강점
- [구체적 예시가 있는 강점]
- [구체적 예시가 있는 강점]

### 개발이 필요한 영역
- [구체적이고 실행 가능한 가이드가 있는 영역]
- [구체적이고 실행 가능한 가이드가 있는 영역]

### 목표 달성
| 목표 | 평가 | 코멘트 |
|------|--------|----------|
| [Goal] | [Rating] | [Specific observations] |

### 영향과 기여
[팀/조직에 대한 가장 큰 기여와 영향을 설명]

### 개발 계획
| 기술 | 현재 | 목표 | 행동 |
|-------|---------|--------|---------|
| [Skill] | [Level] | [Level] | [How to get there] |

### 보상 권고
[승진 / 지분 리프레시 / 조정 / 변경 없음 — 근거 포함]
```

## 출력 - 캘리브레이션

```markdown
## 캘리브레이션 준비: [Review Cycle]

### 팀 개요
| 직원 | 역할 | 레벨 | 재직 기간 | 제안 평가 | 메모 |
|----------|------|-------|--------|-----------------|-------|
| [Name] | [Role] | [Level] | [X years] | [Rating] | [Key context] |

### 평가 분포
| 평가 | 수 | 팀 대비 비율 | 회사 목표 |
|--------|-------|-----------|----------------|
| 기대 초과 | [X] | [X]% | ~15-20% |
| 기대 충족 | [X] | [X]% | ~60-70% |
| 기대 미만 | [X] | [X]% | ~10-15% |

### 캘리브레이션 논의 포인트
1. **[Employee]** — [이 평가가 논의가 필요한 이유, 예: 경계선, 해당 레벨의 첫 리뷰, 최근 역할 변경]
2. **[Employee]** — [Discussion point]

### 승진 후보
| 직원 | 현재 레벨 | 제안 레벨 | 근거 |
|----------|-------------|----------------|---------------|

### 보상 조치
| 직원 | 조치 | 근거 |
|----------|--------|---------------|

### 관리자 메모
[캘리브레이션 그룹이 알아야 할 맥락 - 팀 변화, 조직 변경, 프로젝트 영향]
```

## 연결 도구가 있을 경우

**~~HRIS**가 연결되어 있다면:
- 이전 리뷰 이력과 목표 추적 데이터를 가져옵니다
- 직원 정보와 현재 역할 정보를 미리 채웁니다

**~~project tracker**가 연결되어 있다면:
- 리뷰 기간의 완료 업무와 기여도를 가져옵니다
- 특정 티켓과 프로젝트 마일스톤을 근거로 참조합니다

## 팁

1. **구체적으로 쓰세요** - "잘했어요"는 피드백이 아닙니다. "새 CI 파이프라인을 도입해 배포 시간을 40% 줄였습니다"는 피드백입니다.
2. **긍정과 개선점을 균형 있게** - 둘 다 중요하며, 어느 쪽도 놀라움이 되어서는 안 됩니다.
3. **성격이 아니라 행동에 집중** - "문서가 불완전하다" vs. "당신은 부주의하다"
4. **개발 목표를 실행 가능하게** - "커뮤니케이션을 개선하라"는 모호합니다. "다음 팀 전체 회의에서 발표하라"는 실행 가능합니다.
